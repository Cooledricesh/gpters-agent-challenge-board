# 23기 기술트리 커리큘럼 전면 재설계 — 설계

- 날짜: 2026-07-02
- 배경: 22기 과제는 운영 중 급하게 추가되어 체계가 없었다. 23기는 문명류 게임의
  기술트리처럼 선행 과제 → 파생 과제 구조로 재설계한다. 아울러 사례글 매핑이 과제
  "제목"만 보고 이루어져 오분류가 있었다(예: Honcho/Mem0 메모리 백엔드 과제인
  "기억 보조장치"에 옵시디언 지식저장소 사례가 섞임). 재매핑은 과제 설명 기반으로 한다.

## 확정된 결정사항

1. **전면 재설계**: 사례 98건 분석 + 기존 43개 과제(설명 포함)를 재료로 23기 커리큘럼을
   기술트리 네이티브로 새로 설계. 유지할 과제는 이름·설명·위치를 정리, 애매한 과제는
   병합/은퇴. 현재 수강생 0명이므로 완료기록 손실 없음.
2. **소프트 안내**: 선행 미완료여도 체크는 가능. 카드가 흐릿해지고 "선행: OO" 안내만 표시.
   하드 잠금 없음(서버 거부 없음).
3. **티어 = 난이도 = 점수**: T1=1점, T2=1.25점, T3=1.5점. 기존 basic/advanced 가중치를
   티어 가중치로 대체. `level` 컬럼은 유지(아카이브 호환)하되 화면·점수는 tier 기준.
4. **통합 뿌리 1개**: "오픈클로 or 헤르메스 설치" 단일 과제가 유일한 뿌리. 모든 과제가
   여기서 파생. 선행조건은 **단일 prerequisite_id**(진짜 트리, DAG 아님) — AND/OR 혼재
   없음. 복합 과제는 핵심 선행 하나만 걸고 나머지는 설명에 언급.
5. 입문 난이도 기준: 처음 오픈클로/헤르메스를 접하는 사람이 일정 수준 이상으로 배우는 데
   필요한 과제를 우선한다.

## 구성 요소

### 1. DB 스키마 (add-only)

```sql
alter table public.challenges add column if not exists tier integer not null default 1;
alter table public.challenges
  add column if not exists prerequisite_id uuid references public.challenges (id) on delete set null;
```

- tier ∈ {1,2,3} check 제약. 뿌리 과제만 prerequisite_id null.
- `area`는 가지(branch) 라벨로 재사용. `level`은 유지하되 tier에서 파생 세팅
  (T1→basic, T2/T3→advanced)해 하위 호환.
- 적용: Supabase SQL Editor(사용자 실행). `supabase/schema.sql`에도 반영.

### 2. 커리큘럼 설계 파이프라인

1. **분석 워크플로우**: 사례 98건(case-meta.json) + 기존 43과제(설명·detail 포함 덤프)를
   서로 다른 렌즈의 에이전트들이 분석 — (a) 주제 클러스터·빈도, (b) 도구/기술 선행관계
   신호, (c) 입문자 난이도 신호(트러블슈팅·막힘 지점), (d) 기존 과제 커버리지 갭·신규 과제 후보.
2. 메인 컨텍스트에서 **기술트리 초안 종합**: 과제별 title/description/detail/tier/
   prerequisite/branch. 사례가 실증하는 주제를 우선.
3. **비평 워크플로우**: 초안을 입문자 관점·스터디장(운영) 관점·트리 일관성(고아/사이클/
   티어-깊이 정합) 관점에서 병렬 비평 → 수정.
4. **사용자 검토 게이트**: `scripts/data/curriculum-23.json` + 읽기 좋은 트리 문서 제시,
   승인 후 적용.
5. `apply-curriculum.mjs`: challenges 전체 교체(dry-run 기본, --confirm).
   challenge_examples는 FK cascade로 함께 삭제됨 → 재매핑으로 재적재.

### 3. 설명 기반 재매핑

- 매핑 워크플로우 입력에 과제의 **description + detail 전문** 포함(지난번 오분류 원인 제거).
- 검증 프롬프트에 "과제 설명이 정의하는 완료 기준에 사례가 부합하는가"를 명시.
- 산출 → merge → **사용자 검토** → import (기존 스크립트 파이프라인 재사용:
  prep-mapping / merge-mapping / import-case-studies).

### 4. UI — 기술트리 표시

- `/my` 체크리스트·공개 홈 리스트: 가지별 섹션 안에서 티어 순 정렬, `T1/T2/T3` 배지,
  "🔗 선행: {과제명}" 표시.
- `/my`에서 선행 미완료 과제는 카드 흐림(opacity) + 안내. 체크는 가능.
- 상세 모달: 선행 과제명 + 선배 사례(재매핑본).
- 점수: `challengeWeight(tier)` 및 고정 문구(1점/1.25점) 갱신. progress.ts의 가중치
  헬퍼를 티어 기반으로 확장하고 level 기반 헬퍼는 아카이브 페이지 전용으로 유지.

### 5. 구현 순서

① 스키마 마이그레이션 준비·적용 → ② 분석 워크플로우 → ③ 트리 초안 종합 + 비평 →
**검토 게이트** → ④ 커리큘럼 적용(--confirm) → ⑤ 설명 기반 재매핑 워크플로우 →
**검토 게이트** → ⑥ 사례 import → ⑦ UI + 테스트 + 빌드 → ⑧ 머지·푸시.

## 범위 밖 (YAGNI)

- 하드 잠금, 다중 선행조건(OR/AND 혼합), 노드-엣지 그래프 시각화(티어드 리스트로 충분).
- 관리자 화면 트리 편집 UI(스크립트로 관리).
- 22기 아카이브 페이지 변경(정적 스냅샷 그대로).

## 테스트

- 트리 정합성 검증(사이클 없음, 뿌리 1개, tier-깊이 정합)을 apply 스크립트와 단위 테스트에.
- challengeWeight(tier)·정렬·선행 표시 로직 단위 테스트.
- 공개 홈/체크리스트 렌더 테스트(선행 배지, 흐림 처리).
- npm run build + npm test, 브라우저 육안 확인.
